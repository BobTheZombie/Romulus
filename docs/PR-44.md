# PR 44 - Correct bounded FORUM-family PL8 image dimensions

## Summary

This PR corrects the bounded Caesar II FORUM-family large-PL8 parser so image dimensions are read from
the fixed header as two little-endian `u16` values, then validated against bounded rules derived from
those parsed dimensions.

The parser remains intentionally narrow: it only supports the confirmed 24-byte FORUM-family layout
introduced in PR 43 and fails explicitly for unsupported or inconsistent payloads.

## Header interpretation correction

- Fixed header size remains `24` bytes.
- Parsed dimension fields now use:
  - `width` at offset `8` as little-endian `u16`
  - `height` at offset `10` as little-endian `u16`
- Unknown header fields remain uninterpreted.

This matches observed asset bytes such as `80 02 e0 01`, which decode to `640x480` when interpreted as
two `u16` values.

## Bounded validation rules

Removed the hardcoded `640x480` requirement from the parser.

Supported image validation is now:
- `width > 0`
- `height > 0`
- each dimension must be within bounded range `[1..4096]`
- payload bytes after the fixed 24-byte header must exactly equal `width * height`

If payload size does not match parsed dimensions, parse fails with a deterministic payload mismatch error.

## Reporting updates

Stable PL8-image report output now includes:
- parsed `width` and `height`
- fixed `header_size`
- `payload_size`
- `payload_expected_from_dimensions`
- `payload_matches_dimensions` (`yes`/`no`)

## Integration behavior

Forum composition continues using the same decode path and failure model:
- background decode failure remains fatal for composition
- overlay failures are logged and skipped
- successfully decoded FORUM-family overlays participate in composition

No broad PL8 framework was added in this PR.

## Non-goals

- No generic support for unrelated PL8 variants
- No speculative interpretation of unknown header fields
- No gameplay/map rendering changes
- No ImGui work
