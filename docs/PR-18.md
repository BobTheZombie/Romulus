# PR-18: Win95 Caesar II LBM (ILBM/IFF) Bounded Parser

## Summary

PR-18 introduces a narrow Win95-focused `.LBM` parser for Caesar II `data/*.lbm` assets, targeting IFF `FORM/ILBM` images with explicit bounds checks and structured parse failures.

## What is added

1. **Data-layer ILBM resource and parser**
   - Added `IlbmImageResource` in `src/data/include/romulus/data/ilbm_image.h`.
   - Added `parse_ilbm_image(...)` in `src/data/src/ilbm_image.cpp`.
   - Parser validates and requires:
     - `BMHD` (exact 20-byte header)
     - `CMAP`
     - `BODY`
   - Parser is intentionally narrow:
     - supports only `FORM` type `ILBM`
     - supports only `8` bitplanes (`nPlanes=8`)
     - supports masking mode `0` only
     - supports compression mode `0` (none) and `1` (ByteRun1)
     - fails explicitly on unsupported layouts/chunk duplication

2. **Bounded extraction and decode**
   - Extracts:
     - width/height
     - palette entries from `CMAP`
     - raw `BODY` payload
     - decoded indexed pixels from planar ILBM data
   - All parsing is `BinaryReader`-based and bounded.
   - FORM and chunk sizes are validated for truncation/layout errors.

3. **Stable report helper**
   - Added `format_lbm_report(...)` for repeatable reverse-engineering/debug output.

4. **CLI support**
   - Added `--probe-lbm <path>` mode to `apps/caesar2/main.cpp`.
   - The mode loads one file, parses ILBM, and prints the stable report.

5. **Tests**
   - Added `tests/data/ilbm_image_tests.cpp` with synthetic IFF fixtures covering:
     - valid decode
     - missing required chunk
     - invalid FORM type
     - unsupported bitplane count
     - truncated payload
     - stable report formatting text

## Scope constraints honored

- Parsing remains narrow and asset-driven for Win95 Caesar II data layout.
- No broad multimedia framework introduced.
- No rendering architecture changes introduced.
- No simulation logic added.
