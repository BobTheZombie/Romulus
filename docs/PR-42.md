# PR 42 - Hardcoded Forum Screen Composition Path

## Summary
This PR adds a narrow runtime composition path for the Caesar II forum screen by layering three hardcoded assets in this order:

1. `DATA/FORUM.LBM` (background)
2. `DATA0/FORUM.256 + DATA0/FORUM.PL8` (first overlay)
3. `DATA0/RAT_BACK.256 + DATA0/RAT_BACK.PL8` (second overlay)

The implementation intentionally avoids introducing a generic scene/UI system.

## Asset selection and decode path
- Background uses the existing bounded ILBM parse/decode path.
- Overlay layers use existing bounded `.256` + `.PL8` decode with known dimensions.
- The runtime reuses case-insensitive data-root resolution for all paths.

## Hardcoded layering behavior
- Runtime now attempts the forum composition mode first when no explicit debug image is provided.
- Overlays are decoded with index `0` treated as transparent, then composited directly on top of the currently composed image.
- Composition is full-frame and order-dependent; there is no layout system.

## Failure behavior
- If `FORUM.LBM` cannot resolve/load/parse/decode, runtime preserves the prior startup failure/fallback behavior.
- If an overlay layer fails to resolve/load/decode (or has an unexpected size), runtime logs a warning and continues rendering lower layers.
- Runtime remains stable and exits cleanly on window close / Escape as before.

## Logging
Compact logs were added for each forum layer stage:
- resolved
- loaded
- decoded
- composed
- skipped/failed

## Explicit non-goals in this PR
- No generic scene framework
- No gameplay systems
- No map rendering
- No ImGui integration
- No attempt to fully decode every forum-related asset
