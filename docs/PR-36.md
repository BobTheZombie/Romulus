# PR 36 - Bounded PACK palette-like (`PL8`) entry extraction

## Scope
This PR adds support for exactly one additional Win95 `PACK` entry family: **simple palette-like entries that match the already-supported standalone `.PL8` layout** (exactly 768 bytes / 256 RGB triplets).

Added behavior:
- bounded extraction helper for a `PACK` entry by index that targets this family only
- strict candidate gating (`size == 768` with conservative class filtering)
- reuse of existing `parse_pl8_resource` and `format_pl8_report` paths
- narrow CLI probe/export commands for this family
- focused tests for success and failure behavior

## Why this family first
The simple `.PL8` layout is already implemented and bounded in Romulus. Reusing that parser avoids speculative decoding and keeps this PR low-risk while expanding PACK coverage beyond ILBM/text families.

## Implementation notes
- Added `extract_win95_pack_pl8_entry(...)` to `win95_data_container`.
- Validation remains deterministic and bounded:
  1. entry index bounds check
  2. conservative family gate (must look like supported palette payload family)
  3. offset/size range check against provided container bytes
  4. parse via existing `parse_pl8_resource` (no duplicate decode logic)
- Errors are explicit for:
  - invalid index
  - unsupported/non-PL8-like entry family
  - range overflow / out-of-bounds extraction
  - PL8 parse failure path

## CLI
Added:
- `--probe-pack-pl8 <container> <entry_index>`
- `--export-pack-pl8 <container> <entry_index> --export-output <path>`

Both commands are wired into existing strict mutual-exclusion/argument validation logic.

## Non-goals (explicit)
- no support for multiple PL8 variants
- no speculative archive-wide palette heuristics
- no gameplay semantics
- no ImGui changes
- no weakening of PR 29/30 structural and bounded-analysis behavior
