# PR 39 - Case-insensitive Win95 path resolution for bootstrap assets

## Why this is needed

Caesar II's Win95 distribution uses case-insensitive filesystem semantics, while Linux filesystems are typically case-sensitive. Romulus PR 38 introduced a minimal bootstrap path that loads startup ILBM files from logical paths such as `data/forum.lbm`. Real Win95 installs commonly store these assets as uppercase entries like `DATA/FORUM.LBM`, which causes literal path lookups to fail on Linux.

This PR adds a bounded case-insensitive resolver so Romulus can resolve Win95-style logical paths against host filesystems without requiring users to rename files.

## Resolver scope

A reusable helper was added in the data layer:

- `romulus::data::resolve_case_insensitive(root, relative_path) -> optional<filesystem::path>`

Behavior:

- Accepts relative Win95-style paths containing `\` and/or `/` separators.
- Normalizes separators and resolves segment-by-segment under the configured root.
- Performs case-insensitive matching at each segment and preserves real on-disk casing in the returned path.
- Rejects malformed and unsafe relative paths (absolute paths, root-qualified paths, `.` and `..` traversal segments, and colon-qualified segments).
- Keeps traversal bounded to exactly the requested path depth; no global recursive searching.

## Integration points in this PR

- Bootstrap startup asset lookup now routes relative candidates through the case-insensitive resolver.
- Existing candidate order is unchanged (`data/forum.lbm`, `data/empire2.lbm`, `data/fading.lbm`).
- Override paths are also resolved through the helper when relative.
- Bootstrap logging/error messages now include requested path details, whether case-insensitive resolution was attempted, and the resolved on-disk path when available.

## Non-goals

- No broad virtual filesystem abstraction.
- No global refactor of all game-data call sites.
- No gameplay, map, rendering, or unrelated runtime changes.
- No unbounded filesystem scanning.
