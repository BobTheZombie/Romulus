# PR 43 - Bounded FORUM-style large-PL8 image support

## Summary
This PR adds a narrowly-scoped decoder for the confirmed large-PL8 forum overlay layout:

- `DATA0/FORUM.PL8` is treated as indexed image data in a fixed FORUM-style layout.
- `DATA0/FORUM.256` is treated as the 768-byte palette.

Supported PL8 image layout in this PR is intentionally strict:
- fixed 24-byte header
- width/height read from header
- width/height must be exactly `640x480`
- payload after header must be exactly `width*height` bytes

No generic large-PL8 framework is introduced.

## New bounded parser
Added `Pl8ImageResource` parse/decode path for FORUM-style PL8 images:
- `parse_caesar2_forum_pl8_image(...)`
- `decode_caesar2_forum_pl8_image_pair(image_pl8, palette_256, ...)`
- `format_pl8_image_report(...)`

Decode uses existing indexed image -> palette -> RGBA conversion, with bounded checks and deterministic failures.

## CLI additions
Added new command modes:
- `--probe-pl8-image <path>`
- `--view-pl8-image-pair <imagepl8> <palette256>`
- `--export-pl8-image-pair <imagepl8> <palette256> --export-output <path>`

These modes are mutually exclusive with existing tile/LBM and `.256+.PL8` export/view modes.

## Forum composition fix
Forum overlay composition now uses corrected roles:
- image: `DATA0/FORUM.PL8`
- palette: `DATA0/FORUM.256`

The secondary overlay is also attempted as PL8-image + .256-palette (`RAT_BACK.PL8 + RAT_BACK.256`).
If it does not match the strict FORUM-style layout, decode fails explicitly and the layer is skipped with warning logs.

## Tests
Added/updated tests for:
- valid synthetic FORUM-style PL8 image parse
- invalid short header failure
- width/height payload mismatch failure
- valid PL8-image + 768-byte `.256` palette pairing
- stable report formatting
- bootstrap overlay asset role/order expectations
