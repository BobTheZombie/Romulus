# PR 50: Forum Sprite-Table Overlay Placement

## Why this change exists

`RAT_BACK.PL8` is a sprite-table resource, not a full-screen image. Its sprite descriptors carry placement coordinates (`x`, `y`) that describe where each decoded sprite should appear on the forum screen. Treating the composed sprite-table bounds as if they must equal `FORUM.LBM` dimensions incorrectly rejects valid overlays (for example, a decoded span like `361x412` against a `640x480` background).

## Behavior implemented

- Keep existing decode paths intact:
  - FORUM-style image decode (`FORUM.PL8` + `FORUM.256`) is unchanged.
  - PL8 sprite-table parsing and type-0 sprite decode are unchanged.
  - RAT_BACK structured fallback decode remains unchanged.
- For sprite-table forum overlays, composition now uses a full forum-sized transparent target canvas (`640x480` at runtime, matching the decoded forum background dimensions).
- Decoded sprites are blitted into that full-size canvas using descriptor coordinates, in descriptor order.
- Transparency is preserved by honoring alpha=0 pixels during copy.

## Bounds and clipping policy

This PR chooses **safe clipping** for deterministic bounded behavior:

- Sprite regions that partially intersect the target canvas are clipped to in-bounds pixels and drawn.
- Sprite regions with no intersection are skipped.
- Composition order remains descriptor order.

Runtime logs now report compact placement stats:

- decoded sprite count
- target canvas dimensions
- placed sprite count
- clipped sprite count
- skipped out-of-bounds sprite count

## Non-goals

- No new parser families.
- No UI/layout framework changes.
- No gameplay or map rendering changes.
- No changes to existing PL8 sprite-table decode semantics beyond forum-layer placement behavior.
