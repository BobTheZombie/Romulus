# PR 45 - Reconnaissance/reporting for second large-PL8 variant (RAT_BACK-style)

## Summary

This PR adds a bounded reconnaissance path for Caesar II Win95 large-PL8 image files that share the
FORUM-family fixed header but do not match the simple `width * height` payload layout.

The new path is intentionally reporting-only for the second variant (e.g. `DATA0/RAT_BACK.PL8`).
It does **not** claim decode support for this variant and does not alter existing FORUM-style decode behavior.

## What was added

- Reused shared large-PL8 header parsing (`24`-byte fixed header, dimensions at offsets `8/10`) for both:
  - existing FORUM-style parser
  - new variant probe/report path
- Added deterministic probe report fields for second-variant reconnaissance:
  - parsed width/height
  - fixed header size and payload offset
  - file size and payload bytes after header
  - expected bytes from `width * height`
  - payload surplus/deficit relative to FORUM-style expectation
  - compact payload previews (prefix/suffix)
  - compact extra-payload previews (prefix/suffix when surplus exists)
  - deterministic row-stride/padding hints where derivable from exact arithmetic
- Added deterministic comparison reporting for two probed large-PL8 files (e.g. FORUM vs RAT_BACK)
  including payload deltas and per-file probe blocks.

## CLI additions

Two narrow command modes were added:

- `--probe-pl8-image-variant <path>`
  - probes one large-PL8 image and prints the reconnaissance report.
- `--compare-pl8-image-variants <lhs_imagepl8> <rhs_imagepl8>`
  - probes both files and prints a deterministic comparison report.

These modes are integrated with existing command-mode exclusivity rules.

## Tests added/updated

`tests/data/pl8_image_resource_tests.cpp` now covers:

- stable variant probe formatting
- deterministic surplus reporting for a RAT_BACK-style mismatch fixture
- deterministic comparison output behavior
- malformed/truncated header failure behavior for the variant probe path

## Non-goals (unchanged)

- No generic PL8 framework
- No speculative decode implementation for the second variant
- No gameplay/UI-system changes
