# PR-51: Bounded RAT_BACK placement-semantics analysis

## Summary
This PR adds bounded, deterministic tooling for **placement semantics analysis** of RAT_BACK-style PL8 sprite-table overlays.

The decode/parsing path is unchanged. The new work is focused on interpreting descriptor coordinates (`x`, `y`) and draw order for forum composition debugging.

## Supported placement modes
The runtime and CLI now support these explicit placement modes:

1. `top_left`: destination `(x, y)`
2. `bottom_left`: destination `(x, y - height)`
3. `centered`: destination `(x - width/2, y - height/2)`
4. `bottom_center`: destination `(x - width/2, y - height)`

These modes are intentionally bounded and fixed.

## Runtime debug controls (forum bootstrap composition)
When forum composition includes a sprite-table overlay (for RAT_BACK-style data), debug controls are enabled:

- `1` / `2` / `3` / `4`: select placement mode
- `[` / `]`: isolate previous/next sprite index
- `0`: show all sprites
- `r`: toggle draw order forward/reverse

The runtime logs deterministic placement reports including sprite index, dimensions, descriptor coordinates, tile type, and computed destination rect.

## CLI probe helper
A new probe command was added:

- `--probe-pl8-sprite-placement <imagepl8> <palette256>`

This command decodes sprite-table overlays and prints placement reports across all supported modes.

## Why this PR is about semantics, not decoding
The existing decode path already successfully decodes RAT_BACK sprite payloads. The remaining mismatch appears to be descriptor placement interpretation and/or ordering, so this PR isolates placement interpretation as the variable under test.

## Non-goals
- No parser/decode format expansion
- No generalized UI/layout framework
- No gameplay/map rendering
- No ImGui/editor tooling
- No freeform transforms or arbitrary runtime placement editing
