# PR 31 - Bounded ILBM extraction from Win95 PACK containers

## Scope
This PR adds a narrow, bounded extraction path for exactly one PACK entry family: entries already classified as `possible-ilbm` by PR 30.

Implemented capabilities:
- extract PACK entry bytes by index from a parsed PACK container
- enforce index and payload bounds during extraction
- require `possible-ilbm` entry classification before decode
- validate decoded payload with the existing ILBM parser
- reuse existing ILBM report/export/view pipelines
- expose focused CLI modes for probe/export/view of PACK ILBM entries

## Why ILBM first
ILBM was chosen as the first extracted family because:
- existing parser and conversion infrastructure already exists (`parse_ilbm_image`, `convert_ilbm_to_rgba`)
- PACK heuristic classification from PR 30 already identifies likely ILBM candidates (`possible-ilbm`)
- this allows low-risk, bounded end-to-end extraction without introducing speculative archive semantics

## Reuse of existing ILBM pipeline
The PACK extraction helper only slices and validates a bounded payload, then delegates all image parsing and decode to existing ILBM code paths:
- parser: `parse_ilbm_image`
- report: `format_lbm_report`
- conversion: `convert_ilbm_to_rgba`
- export: existing RGBA export path
- viewer: existing SDL debug viewer path via `Application`

No ILBM decode logic is duplicated in PACK-specific code.

## Explicit failures
The extraction path now returns explicit failures for:
- invalid entry index
- entry family unsupported (`classification_hint != possible-ilbm`)
- truncated/invalid payload bounds
- malformed ILBM payload
- export/view failures after decode

## Non-goals
This PR intentionally does **not**:
- add generalized extraction for all PACK entries
- infer gameplay semantics of extracted payloads
- alter PR 29 structural PACK validation logic
- weaken PR 30 bounded entry analysis/reporting behavior
- introduce ImGui
- create a broad archive framework
